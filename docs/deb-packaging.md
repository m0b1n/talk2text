# Debian Packaging Notes

This document describes what was added to this repository to produce an Ubuntu `.deb` package for the Talk2Text desktop app, why each piece exists, and what was required to get the packaging flow working end to end.

## Goal

The goal was to turn this repository from a Python/PySide6 project into something that can be installed on Ubuntu like a normal desktop application:

- build a runnable bundled app
- place it into the standard Linux filesystem layout
- add launcher metadata and an icon
- wrap the result into a `.deb`

The final output of this work is a Debian package such as:

```text
talk2text_0.1.0-1_amd64.deb
```

## Packaging Strategy

For this repository, the practical strategy is:

1. Use `pyside6-deploy` to bundle the PySide6 app into a Linux runnable distribution.
2. Use a repo-owned shell script to stage the bundled output into a Debian package layout.
3. Use `dpkg-deb` to build the final `.deb`.

This is a bundle-first approach. It is more suitable for this app than pure Debian-native Python packaging because the project depends on PySide6, faster-whisper, and related runtime pieces that are easier to ship as a bundled app than as a collection of Ubuntu package dependencies.

## Files Added Or Updated

The following files were added or changed to support Debian packaging.

### `build-deb.sh`

This is the main packaging helper script.

It does the following:

- reads the project version from `pyproject.toml`
- runs `pyside6-deploy`
- collects the generated bundled app output
- creates a Debian package root under `build/pkgroot`
- installs the app bundle under `/opt/talk2text`
- installs a wrapper launcher under `/usr/bin/talk2text`
- installs the desktop entry under `/usr/share/applications/`
- installs the icon under `/usr/share/icons/hicolor/scalable/apps/`
- generates the `DEBIAN/control` file
- runs `dpkg-deb --root-owner-group --build`

### `pysidedeploy.spec`

This is the deploy configuration used by `pyside6-deploy`.

It tells the deploy tool:

- where the project root is
- which Python file should be treated as the app entrypoint
- which Python interpreter to use
- which build-time helper packages must exist
- which Qt plugins/modules should be included
- which Nuitka mode and extra arguments should be used

### `packaging/main.py`

This file exists specifically for deployment.

It acts as a small deployment entrypoint that:

- resolves the repo `src/` directory
- imports `talk2text.ui.main`
- starts the application

This keeps the deployment entrypoint simple and explicit for `pyside6-deploy`.

### `packaging/io.github.m0b1n.Talk2Text.desktop`

This is the desktop launcher metadata.

It makes the app install like a normal desktop program by declaring:

- the application name
- the installed command
- the icon name
- categories and keywords
- desktop launcher behavior

### `packaging/io.github.m0b1n.Talk2Text.svg`

This is the application icon used by the desktop entry and the packaged app.

### `.gitignore`

This was updated so packaging outputs do not get committed accidentally.

The important ignored outputs are:

- `build/`
- `deployment/`
- `packaging/deployment/`
- `*.deb`

### `README.md`

This was updated so the repository documents:

- how to build the `.deb`
- that `patchelf` is required for standalone Linux bundling
- how to install the generated package

## Debian Package Layout

The build script creates a package root with this structure:

```text
build/pkgroot/
  DEBIAN/
    control
  opt/talk2text/
    ...bundled app files...
  usr/bin/
    talk2text
  usr/share/applications/
    io.github.m0b1n.Talk2Text.desktop
  usr/share/icons/hicolor/scalable/apps/
    io.github.m0b1n.Talk2Text.svg
```

This layout is intentional:

- `/opt/talk2text/` contains the bundled runtime and app files
- `/usr/bin/talk2text` is the command users launch
- `/usr/share/applications/...desktop` integrates with the Ubuntu app menu
- `/usr/share/icons/...` provides the app icon to the desktop environment

## Wrapper Launcher

The package installs a small shell wrapper at:

```text
/usr/bin/talk2text
```

Its job is to execute the actual bundled binary from `/opt/talk2text/...`.

This separation is useful because:

- users get a stable command name
- the internal bundle path can remain private to the package
- the desktop entry can simply call `/usr/bin/talk2text`

## `DEBIAN/control`

The packaging script generates the Debian control file dynamically at build time.

It includes:

- package name
- version
- architecture
- maintainer
- basic runtime dependencies
- package description

The control file is generated by the script rather than committed as a static file so the package version can be pulled directly from `pyproject.toml`.

## Build Requirements

The packaging flow required these pieces to exist on the machine:

- a Python virtual environment under `.venv`
- project dependencies installed in that environment
- `pyside6-deploy`
- Nuitka and related deploy-time Python packages
- the system `patchelf` binary
- `dpkg-deb`

In practice, the important commands are:

```bash
python3 -m venv .venv
.venv/bin/pip install -e .
sudo apt install patchelf
./build-deb.sh
```

## What Was Necessary To Make It Work

Several packaging-specific issues had to be resolved before the `.deb` build worked.

### 1. Add a dedicated deployment entrypoint

`pyside6-deploy` works best when given a simple Python entry file. The normal package entrypoints were fine for development, but a dedicated `packaging/main.py` made deployment more explicit and stable.

### 2. Add desktop metadata

A `.deb` for a desktop app is not complete without a `.desktop` file and icon. Those were necessary for the installed package to behave like a normal Ubuntu application.

### 3. Add a scripted package assembly step

`pyside6-deploy` bundles the application, but it does not directly produce the final Debian package layout we wanted for this repository. That is why `build-deb.sh` was needed.

### 4. Make the deploy config portable

The deploy config needed to avoid hardcoded machine-specific absolute paths where possible. This matters so the repository can be reused on another machine without rewriting committed packaging files.

### 5. Fix the `packages` format in `pysidedeploy.spec`

`pyside6-deploy` expects the `packages` field to be comma-separated, not shell-style space-separated.

This had to be corrected:

- wrong: `nuitka==2.8.4 ordered_set zstandard patchelf`
- right: `nuitka==2.8.4,ordered_set,zstandard,patchelf`

### 6. Fix the Nuitka extra arguments

The `--noinclude-qt-translations` flag was initially written in a form Nuitka rejected. It had to be corrected to the syntax Nuitka actually accepts.

### 7. Install the system `patchelf`

Even though the Python `patchelf` package was installed by `pyside6-deploy`, Nuitka standalone Linux builds still required the actual system `patchelf` binary to be available on the machine.

This was a real external prerequisite for the standalone Linux bundle.

### 8. Match the deploy output path used by `pyside6-deploy`

The build script originally expected the deployment output in the wrong place. The script had to be aligned with the actual output directory produced by the tool during real execution.

### 9. Detect the real app binary correctly

The script originally searched too broadly for `*.bin` files and accidentally picked up build artifacts inside the Nuitka build tree.

This had to be narrowed so it only looks for the packaged application binary under the actual `*.dist/` directory.

Without that fix, the bundle compiled correctly, but the package assembly step still failed.

## Build Flow

The working build flow is now:

```bash
chmod +x build-deb.sh
./build-deb.sh
```

Internally, that performs:

1. `pyside6-deploy` bundles the app
2. the bundle is copied into `build/pkgroot/opt/talk2text/`
3. launcher and icon files are installed into package locations
4. `DEBIAN/control` is written
5. `dpkg-deb` creates the final `.deb`

## Result

The packaging setup was verified by producing a real package file:

```text
talk2text_0.1.0-1_amd64.deb
```

The package can be installed with:

```bash
sudo apt install ./talk2text_0.1.0-1_amd64.deb
```

## What This Does Not Yet Cover

This packaging setup produces a valid installable Debian package for the app, but it does not yet include:

- maintainer scripts such as `postinst` or `prerm`
- MIME type registration
- automatic Ollama installation
- repository publishing
- AppArmor or Snap packaging
- Debian policy-level refinements beyond the current practical installer flow

Those can be added later if needed, but they were not necessary to get a working Ubuntu `.deb` build for this project.
